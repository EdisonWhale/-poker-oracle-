你现在是一个“产品 + 技术架构 + UI/UX + Agent 团队编排”的资深负责人。请为我准备一份**可落地的详细设计报告**，目标是做一个**网页端德州扑克（Texas Hold’em）训练型平台**。

## 0) 背景与目标（你需要先复述确认）

* 这不是传统“约朋友来玩”的德州平台，更像是**个人训练场**。
* 用户可以**自己创建房间**，然后选择加入：

  * 先期：基础“人机（规则/策略驱动 bot）”与“练习模式”
  * 后期：可扩展到 ML 训练的机器人、以及接入不同 LLM/AI 玩家（例如 GPT / Gemini / Claude）作为对手或教练
* 我当前阶段优先级：**先把游戏整体做出来**（核心玩法 + UI/交互 + 房间系统 + 回放/训练功能），AI/ML 放到后续里程碑。

---

## 1) 交付物要求（请按章节输出）

请给我一份结构化报告，至少包含：

### A. 产品范围与核心体验（MVP 优先）

* 定义 2–3 个核心使用场景（例如：新手训练、针对某类局面的练习、快速开一局）
* MVP 功能列表（Must / Should / Could 分级）
* 关键指标（比如：局数、胜率、摊牌回放观看率、训练任务完成率）

### B. UI/UX 设计（前期重点，尽量具体）

* 信息架构：有哪些页面/模块（例如首页、创建房间、牌桌、训练/任务、回放、个人资料、设置）
* 牌桌界面布局：玩家位、筹码、底池、行动按钮、计时、手牌/公共牌展示、行动历史、快捷键
* 训练向功能的 UI：提示/建议、复盘标注、错误提示、难度选择、推荐训练任务
* 多端适配策略（桌面优先还是移动优先？响应式怎么做）

### C. 技术架构与 Tech Stack（务必给出取舍理由）

* 前端（推荐栈 + 备选栈）：渲染牌桌动画、状态管理、WebSocket 通信
* 后端：房间/对局状态机、玩家动作校验、发牌/洗牌 RNG、公平性与防作弊设计
* 实时通信：WebSocket / WebRTC 的选择与原因
* 数据存储：用户、对局、回放、训练任务、bot 策略配置
* 部署与运维：本地开发 → staging → production，日志、监控、告警
* 可扩展性：未来接入 ML/LLM bot 时，如何隔离“对局引擎”和“AI 决策服务”

### D. 游戏引擎与规则实现（核心系统）

* 德州规则与关键边界（盲注、按钮位、边池 side pot、all-in、多玩家摊牌）
* 对局状态机（从开局到结算的状态流转）
* 动作校验与回滚策略（断线重连、超时弃牌、并发动作处理）
* RNG 与可验证性（例如 server-seed / client-seed，可选）

### E. Bot / AI 路线图（先规划，但不强行落地）

* Phase 1：规则/启发式 bot（紧凶/松凶/随机等），用于快速填充对手
* Phase 2：可配置策略 bot（基于 hand ranges / 简化 CFR / ICM 近似等）
* Phase 3：ML bot（自博弈/强化学习）与 LLM 玩家（“人类风格”、解释型教练）
* 明确每一阶段需要的接口（决策 API、观察空间、动作空间、延迟/超时处理、安全沙箱）

### F. 开发计划与 Agent Team 分工（你需要“组建一个团队”）

请把任务拆成 4–6 个 Agent（角色），每个 Agent 输出：

* 负责范围
* 关键设计决策
* 交付物（文档/原型/接口定义/代码模块）
* 风险点与验证方式

建议角色（可调整）：

1. Product + UX Agent
2. Game Engine Agent
3. Backend + Realtime Agent
4. Frontend Table UI Agent
5. Data + Replay/Analytics Agent
6. Bot/AI Roadmap Agent

### G. 风险清单与优先级

* 技术风险（实时一致性、边池复杂度、作弊、断线重连）
* 产品风险（训练功能不够“可感知价值”）
* 范围控制（哪些功能坚决不做进 MVP）

* 先支持桌面网页（移动端后续）
